1. Executive Summary

The Office of School Safety (OSS) is implementing a structured, practical approach to improving the accessibility of its public-facing web content.

This work focuses on what users can access, understand, and use today, while creating a sustainable path for improving legacy content over time.

Content is classified as either:

  • Current → available for active use and required for completing tasks.
  • Archived → retained for reference and clearly identified as historical.

Archived content remains available to users and is clearly identified as historical. It is not presented as current or required for accessing services.

Given the volume of existing materials, this approach prioritizes:

  • Content that supports current services and decision-making.
  • Clear identification of historical materials.
  • A consistent method for providing accessible versions upon request.

This work supports compliance with the Colorado Web Accessibility Act (HB21-1110) and the U.S. Department of Justice ADA Title II rule.

OSS is operating on an accelerated timeline ahead of the federal ADA Title II deadline of April 24, 2026.

Accessibility improvements are guided by a structured, data-informed approach. While Siteimprove is used to monitor HTML web pages, document accessibility work is prioritized using an inventory of documents and available usage data.

2. Legal Framework

Colorado Web Accessibility Act — HB21-1110

State accessibility standards are fully enforceable as of July 1, 2025. Failure to meet these standards constitutes a civil rights violation under state law.

OSS provides a public accessibility statement and request pathway through the Accessibility Resources page. As required by OIT Rule 11.6, this statement includes a commitment to a timely response and provides multiple contact methods.

ADA Title II — DOJ Rule

The DOJ Rule establishes WCAG 2.1 Level AA as the enforceable technical standard. Public entities must conform to this standard for all active web content and mobile applications by April 24, 2026.

Critical compliance boundary

Historical materials may be retained as archived content when they are kept for reference, are not part of current programs or services, and are not used to complete user tasks. Archived content must be clearly identified and supported through the accessibility request process.

3. Content Strategy

Overview

This phase establishes how PDF content will be handled across the OSS website to support accessibility and enable efficient progress within the available timeline.

Some PDF content cannot be remediated within current timelines. These documents are placed in Archived status so work can continue without delay. Archived content may be remediated later and returned to Current status when it meets accessibility requirements.

The approach prioritizes clarity for users and consistency in decision-making.

Core Model

Core Rule: If content supports a current task, decision, or action, classify it as Current. If it does not, classify it as Archived.

  • Current → supports active use and is prioritized for accessibility improvement.
  • Archived → retained for reference, clearly labeled, and supported through the accessibility request process.

This approach prioritizes access to current information while maintaining availability of historical materials.

Preferred Technical Implementation: Automated Boolean Ticketing Workflow

To maximize efficiency and reduce human error, OSS utilizes a structured database approach for archiving content. Because the site is hosted on a vendor-managed Drupal platform (Tyler Technologies), the Webmaster must submit a formal development ticket to establish the architecture.

Step 1: The Vendor Ticket Request

The Webmaster will submit a ticket to Tyler Technologies requesting the following three deliverables:

  1. Database Field: Add a boolean checkbox on Document/Media entities within Drupal (Requested Machine Name: field_is_archived).
  2. Automated Trigger: When checked, Drupal must automatically append the text "(Archived)" to the front-end link output.
  3. Event Interception: Build a global JavaScript listener using the native HTML5 <dialog> element to trap focus. It must detect clicks on these archived links, utilize event.preventDefault(); to pause the document load, and trigger the accessible Gateway modal containing OSS-approved disclaimer text.
Ticket Reference Code for Tyler Technologies

The Webmaster should provide the following optimized HTML/JS logic to TylerTech within the support ticket to ensure native focus-trapping and semantic accessibility standards are met:

<!-- Native HTML5 Gateway Modal Template -->
<dialog id="archiveModal" class="modal-dialog" aria-labelledby="modalTitle">
    <h2 id="modalTitle" class="modal-title">Historical Document Archive</h2>
    <p>This document is historically archived. It has not been updated for WCAG 2.1 AA compliance. 
    It is retained for reference purposes only and is not required for current state services.</p>
    <p>To request an accessible version of this document within 3 to 5 business days, please use our accommodation form.</p>
    <div class="modal-actions">
        <a href="https://oss.colorado.gov/accessibility" class="btn btn-primary">Request Accessible Copy</a>
        <a href="#" class="btn btn-secondary" id="btnProceed" target="_blank">Proceed to PDF</a>
        <button type="button" class="btn btn-secondary" id="btnCancel">Cancel</button>
    </div>
</dialog>

<!-- Interceptor Script -->
<script>
    const modal = document.getElementById('archiveModal');
    const btnProceed = document.getElementById('btnProceed');
    const btnCancel = document.getElementById('btnCancel');
    let lastFocusedElement; 

    document.querySelectorAll('.archived-pdf-gateway').forEach(link => {
        link.addEventListener('click', function(e) {
            e.preventDefault(); 
            lastFocusedElement = document.activeElement; 
            btnProceed.href = this.href; 
            modal.showModal(); 
        });
    });

    function closeModal() {
        modal.close();
        if (lastFocusedElement) lastFocusedElement.focus(); 
    }

    btnCancel.addEventListener('click', closeModal);
    btnProceed.addEventListener('click', closeModal);
</script>

Step 2: Content Execution

Once Tyler Technologies completes the ticket, the Webmaster workflow becomes entirely automated:

  1. The Webmaster filters the Master Inventory spreadsheet for "Archived" PDFs.
  2. The Webmaster logs into Drupal, locates the targeted Media Entities, and simply checks the field_is_archived boolean box.
  3. The backend code handles the compliance routing, ensuring the user reads the disclaimer and is offered an accommodation request path before proceeding to the legacy document.

5. Accessibility Improvement Strategy

OSS prioritizes high-impact remediation and ensuring all new content is authored accessibly in source programs (e.g., MS Word) prior to publication.

6. How the System Works

  1. Inventory: Use the Inventory of Record to identify documents and their locations.
  2. Prioritize: Use available data and document purpose to determine level of use and importance.
  3. Classify: Apply the Current vs Archived decision model using the established workflow.
  4. Apply in Drupal: The Webmaster checks the field_is_archived box for prioritized items.
  5. Remediate: The Accessibility Specialist remediates Priority 1 and 2 documents.

Appendix A: Spreadsheet Decision Workflow

This workflow supports consistent classification of documents using the Inventory of Record and available data.

Core Rule

Core Rule: If content supports a current task, decision, or action, classify it as Current. If it does not, classify it as Archived.

  • Current → Required to complete a task, access a service, or support active use.
  • Archived → Retained for reference and not part of current use.

Decision Criteria

  • Does the document support a current program, service, or requirement?.
  • Would a user need this to complete a task?.
  • Is it used in active workflows or decision-making?.

If the answer is no, the document may be classified as Archived.

Data Support

  • Use spreadsheet classification as the starting point.
  • Use analytics (downloads/usage) to confirm level of use.

Final Decision

  • High use or required → Current (remediate)..
  • Low or no use, reference only → Archived..

If unsure → classify as Archived and continue.

Appendix B: Workflow Steps

Use this quick workflow when reviewing documents in the Inventory Spreadsheet.

Core Rule: If content supports a current task, decision, or action, classify it as Current. If it does not, classify it as Archived.

Step 1: Identify the document

  • Locate the document in the spreadsheet.
  • Confirm document type (form, report, guidance, etc.).

Step 2: Check purpose

  • Is this required to complete a task or access a service?.
  • Is it tied to a current program or requirement?.

Yes → Current

Step 3: Check usage

  • Review available analytics (downloads or interaction).
  • Determine whether the document is actively used.

Low or no use → likely Archived

Step 4: Apply classification

  • Current: Prioritize for remediation.
  • Archived: Mark in Drupal to apply archived status.

Step 5: Move on

  • Do not revisit earlier decisions.
  • Work systematically through the inventory.

One Rule

If it supports current use → Current. Otherwise → Archived.

Appendix C: Data-Informed Prioritization

OSS uses a combination of the Inventory of Record and analytics data to prioritize accessibility work.

Inventory of Record (Primary Source)

  • The spreadsheet serves as the central inventory of documents.
  • Each document is mapped to its web location and server path.
  • Classification decisions originate from this inventory.

Analytics (Supporting Data)

  • Google Analytics is used to understand document usage.
  • Metrics include downloads and user interaction where available.
  • Data helps confirm whether a document is actively used.

How Data is Used

  • High use + high importance → Priority 1.
  • Moderate use → Priority 2.
  • Low or no use → candidate for Archived (Priority 3).

Key Principle

Core Rule: If content supports a current task, decision, or action, classify it as Current. If it does not, classify it as Archived.

Data supports decisions but does not replace them. Final classification is based on whether the content supports current user needs.

Appendix D: Accessibility Improvement Methodology

This appendix defines how accessibility improvements are implemented over time. The approach prioritizes high-impact changes, supports consistent workflows, and focuses effort where it improves access for the greatest number of users.

Overview

Accessibility work is carried out in phases. This ensures that the most impactful issues are addressed first, while supporting sustainable improvement over time.

Phase 1: System-Level Improvements (Highest Impact)

Initial work focuses on issues that affect multiple pages or documents at once. These changes provide the greatest improvement with the least repeated effort.

  • Fix template-level and theme-level accessibility issues.
  • Ensure navigation structure is consistent and usable.
  • Correct labeling issues in forms and interactive elements.
  • Improve link clarity and consistency across the site.
  • Ensure interactive elements meet keyboard and focus requirements.

These changes should be completed before large-scale document remediation begins.

Phase 2: Preventing New Issues

After system-level fixes are in place, the next priority is preventing new accessibility issues from being introduced.

  • Use accessible templates for documents and web content.
  • Create documents accessibly in source formats (e.g., Word) before export.
  • Provide clear guidance for content creators.
  • Avoid publishing new inaccessible PDFs.

Preventing new issues reduces long-term remediation effort and supports a “born accessible” approach.

Phase 3: Targeted Remediation (Current Content)

Remediation efforts focus on content classified as Current (Priority 1 and Priority 2). These documents support active use and must be accessible.

  • Prioritize documents based on usage and importance.
  • Correct structural issues such as tagging and reading order.
  • Provide alternative text for meaningful images.
  • Ensure content can be accessed and understood by screen reader users.

Archived content (Priority 3) is not remediated unless an accessible version is requested.

Phase 4: Monitoring and Continuous Improvement

Accessibility work continues through ongoing monitoring and review.

  • Use analytics to identify high-use content.
  • Track trends in accessibility issues over time.
  • Review workflows and update practices as needed.

This ensures that improvements are maintained and that new issues are identified early.

Verification Approach

Accessibility improvements must be verified using both automated and manual methods.

  • Automated tools may be used for initial checks.
  • Manual review is required for structure, labeling, and usability.
  • Screen reader testing (NVDA or JAWS) is used for high-priority content.

Automated testing alone is not sufficient to confirm accessibility.

Key Principle

Focus effort on current content, prevent new barriers, and apply improvements where they have the greatest impact for users.

Appendix E: Alternative Gateway Implementations

While the Boolean Checkbox approach is the preferred automated method, alternative workflows are recorded here in the event that Tyler Technologies cannot implement the boolean logic on legacy content blocks.

Alternative 1: The CSS Class Method

If PDFs are embedded within WYSIWYG text editors (e.g., CKEditor) where boolean fields cannot be natively applied to specific in-line links, a CSS class method may be used.

  • Workflow: The Webmaster manually edits the HTML link source and appends a specific class (e.g., class="archived-pdf-gateway").
  • Routing: The global JavaScript listener targets this class to execute the event.preventDefault(); command and display the Gateway modal.
  • Drawback: This approach increases human error risk and requires minor HTML editing by content creators.

Alternative 2: Direct Twig Template Logic (No Modals)

If modal windows are determined to be too complex to make fully accessible across all devices, an inline Twig routing logic can be applied directly to the theme.

  • Workflow: When the field_is_archived boolean is checked, the Twig template intercepts the document display.
  • Routing: Instead of a modal, the user is presented with a standard webpage containing the disclaimer and an inline button to proceed.
  • Drawback: This disrupts the visual flow of the page and requires the user to load an entirely new view before accessing the document.
